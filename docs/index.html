The Retrieval Context Protocol

One open protocol so any RAG engine — any language, any vendor — can share its power, and any client can consume it uniformly.

RCP is to retrieval what MCP is to tools and ACP is to agents. Embed, rerank, retrieve, and graph-search across engines with capability negotiation and graceful degradation.

Read the spec → Quickstart

Why RCP

Uniform access

One client speaks to a local embedded index, a hosted vector DB, a Python research engine, and a knowledge graph — without bespoke glue for each.

Capability negotiation

Clients discover what a server can do at connect time and never call an unsupported method. Presence implies support; absence is graceful.

Language & transport free

JSON-RPC 2.0 over newline-delimited stdio or HTTP. If it can read a line and parse JSON, it can be an RCP server. Official SDKs in C++ & Python; the wire is trivial to implement anywhere.

Composable

Servers chain, fuse, and re-export. A gateway can be a server to its clients and a client to upstream engines, fusing their hits.

Forward compatible

A client built for RCP/1 interoperates with any RCP/1+ server using the intersection of their capabilities. Additive changes never break.

Tiny to implement

A conforming server needs one handshake method plus one retrieval capability. The reference server fits on a screen.

How it works

  1. Connect a transport — spawn the server as a subprocess (stdio) or point at an HTTP endpoint.
  2. Initialize — the client sends its version; the server replies with the negotiated version and its capabilities.
  3. Call only the advertised methods: embed, rerank, retrieve, graph, index/*.
  4. Shutdown — or just close the pipe.
// handshake
--> {"jsonrpc":"2.0","id":1,"method":"initialize",
     "params":{"protocolVersion":1,"client":{"name":"app"}}}
<-- {"jsonrpc":"2.0","id":1,"result":{
     "protocolVersion":1,
     "server":{"name":"my-engine","version":"2.3"},
     "capabilities":{"embed":{"dimension":384},
                     "retrieve":{"maxK":100}}}}

// retrieve
--> {"jsonrpc":"2.0","id":2,"method":"retrieve",
     "params":{"query":"eiffel tower","k":3}}
<-- {"jsonrpc":"2.0","id":2,"result":{"hits":[
     {"id":"d1","score":0.91,"text":"…Paris…"}]}}

Quickstart

RCP ships one type-theoretic C++ SDK — invariants proved at compile time (strong scalars, refinement types, a typestate client, a concept-gated server, Result<T> = std::expected, no exceptions for control flow) — plus Python bindings layered over it. Cross-language interop is proven: a Python client talks to a C++ server and back.

Server (Python)

import rcp

s = rcp.Server()
s.set_info("my-engine", "1.0")
s.advertise(rcp.Capability.Retrieve,
            {"maxK": 100, "modes": ["hybrid"]})

@s.on(rcp.Method.RETRIEVE)
def _(params):
    hits = my_index.search(
        params["query"], params.get("k", 10))
    return {"hits": [
        {"id": h.id, "score": h.score,
         "text": h.text} for h in hits]}

s.serve_stdio()

Client (Python)

import rcp

c = rcp.connect_stdio(
        ["python3", "my_engine.py"])
print(c.server, c.capabilities)

if c.supports(rcp.Capability.Retrieve):
    for h in c.retrieve("eiffel tower", k=3):
        print(h["id"], h["score"])

Server (C++)

#include "rcp.hpp"
using namespace rcp;

struct Engine {
  PeerInfo info() const { return {"my-engine","1.0"}; }
  Capabilities capabilities() const {
    return Capabilities{}.with_retrieve({{"maxK",100}});
  }
  Result<Json> retrieve(const Json& p) {
    return Json{{"hits", search(p["query"], p["k"])}};
  }
};

int main() { Server{Engine{}}.serve_stdio(); }

Client (C++)

#include "rcp.hpp"
using namespace rcp;

auto c = Client::connect_stdio({"./my_engine"});
if (!c) return 1;                 // handshake failed

if (auto k = TopK::make(3)) {     // 0 is unrepresentable
  auto hits = c->retrieve("eiffel tower", *k);
  if (hits) for (auto& h : *hits)
    std::printf("%s %.3f\n", h.id.c_str(),
                h.score.get());
}

Prefer another language? RCP is just JSON-RPC over a pipe — the JSON Schema is the normative contract, and the conformance suite validates any server in any language.

Many engines, two models

Have more than one RAG backend? RCP layers both routing models on the core retrieve + capability handshake — zero new server obligations. Both are in the SDK, and both can be driven from a single rcp.json registry.

Selector — pick one

Choose a single backend by id, by required capability, or by priority with liveness fallback (primary → secondary). Connections are lazy: listing five candidates opens zero subprocesses until you select one.

sel = rcp.Selector.load("rcp.json")
c = sel.select_primary()      # highest-priority live
c.retrieve("query", k=5)

Federation — query all

Fan a query out to every engine that supports it, concurrently, then fuse the ranked lists with Reciprocal Rank Fusion — the interoperable default that needs only ranks. Each hit is tagged with its origin engine.

Federation fed;
fed.add("docs", std::move(*docs))
   .add("web",  std::move(*web));
auto hits = fed.retrieve("query", *TopK::make(10));

The agent stack

RCP is designed to sit beside MCP and ACP. An agent runtime speaks ACP to its client, calls tools via MCP, and fetches grounding context via RCP — all sharing JSON-RPC framing and the same initialize/capabilities handshake.
