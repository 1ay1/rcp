## Why RCP

<CardGrid>
  <Card title="Frontier-native" icon="rocket">
    Hybrid dense+sparse search, ColBERT/ColPali late interaction (MaxSim),
    rerankers, MMR diversification, GraphRAG, agentic multi-hop, citations, and
    metadata filtering are all first-class — not bolted on.
  </Card>
  <Card title="Transport-free" icon="random">
    JSON-RPC 2.0 over newline-delimited stdio *or* HTTP. If it can read a line
    and parse JSON, it can speak RCP.
  </Card>
  <Card title="Composable" icon="puzzle">
    A registry (`rcp.json`) or a catalog-capable aggregator lets a client target
    one backend (**Selector**) or fuse many (**Federation**) with RRF.
  </Card>
  <Card title="Type-theoretic SDKs" icon="seti:cpp">
    A header-only C++23 SDK with refinement types (`TopK`, `Dimension`,
    `Score`) plus pybind11 Python bindings. Conformance suite included.
  </Card>
</CardGrid>

## The missing third contract

MCP lets an agent call **tools**. ACP lets an editor drive an **agent**. RCP is
the layer that grounds them both: it lets any client pull **retrieval context**
from any RAG engine over one tiny, vendor-neutral wire.
